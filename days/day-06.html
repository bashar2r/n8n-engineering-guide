Section 01

The Six Postgres Operations

Every workflow you've built ends by handing data to the next system. Today is about the moment that data becomes permanent — written to a table a business actually runs on.

Operation What It Does Risk If Misused
Execute Query Runs any raw SQL you write ($1,$2,$3 for parameters) SQL injection if not parameterized
Insert Adds new rows Duplicate-key error if row already exists
Update Modifies existing rows matching a condition Silently updates zero rows if match fails
Upsert Insert if new, Update if exists (by unique key) Requires correctly defined unique constraint
Select Reads rows matching a condition None (read-only)
Delete Delete rows / Truncate table / Drop table Deletes ALL rows if no condition given
Section 02 — Critical

Parameterized Queries

The single most important production concept in this lesson. This is today's equivalent of Day 2's $json.body and Day 5's Full Response toggle.

Never Build SQL with String Interpolation

Any time user-influenced data ends up inside a SQL string, it must go through $1/$2/$3 and the Query Parameters field — never through direct string interpolation, and never through {{ }} expressions written directly inside the query text.

VULNERABLE — Never Do This
code-node.js
// Code node building a query via
// string interpolation
const email = $json.email;
const query = `SELECT * FROM users
  WHERE email = '${email}'`;

// If email contains:
// ' OR '1'='1

// The query becomes:
// SELECT * FROM users
// WHERE email = '' OR '1'='1'
// → Returns EVERY row. Full breach.

Result: Complete data breach

'1'='1' is always true → returns entire table

SAFE — Always Do This
postgres-node.json
-- Execute Query operation
SELECT * FROM users
WHERE email = $1
Query Parameters
{{ $json.email }}

Result: Safe, no matter the input

$1 is never treated as SQL text — always a literal value

The Rule

The database driver itself enforces separation between code and data. Even if someone submits ' OR '1'='1 as an email, it is searched for as a literal (bizarre) email address, matches nothing, and returns zero rows. Parameterization is the actual fix — not "sanitizing" the string yourself.

Section 03

Query Batching — Database-Level Reality

When a Postgres node receives multiple items, the Query Batching setting controls exactly how those items become database round trips. This maps directly back to Day 1's items-array model.

1

Single Query

Fastest, least granular

-- All N items → 1 combined query
-- 1 round trip to Postgres
✓ Fastest for large item counts
✗ Less granular — one bad item may fail entire batch
2

Independently

Per-item isolation

-- Item 1 → Query 1 → response 1
-- Item 2 → Query 2 → response 2
-- ...
-- Item N → Query N → response N
-- N separate round trips
✓ Per-item success/failure — one bad row doesn't affect others
⚠ Slower for large N (N round trips)
3

Transaction

All-or-nothing atomicity

BEGIN;
  Query 1;
  Query 2;
  ...
  Query N;
COMMIT;   -- only if ALL succeeded
ROLLBACK; -- if ANY failed
✓ Atomic — partial success is impossible
⚠ Slower — trades speed for correctness

Use for: Invoice + line items, financial ledgers — any case where partial writes corrupt data

Production Note

Even "Single Query" batching does not give you true bulk-COPY-level performance. For truly massive loads (tens of thousands of rows), production engineers sometimes use a manually constructed multi-row INSERT ... VALUES (...), (...), (...) via Execute Query, or an external ETL step.

Section 04

Upsert — Insert or Update, Correctly

The Upsert operation requires you to specify which column(s) define uniqueness. Under the hood, this maps to Postgres's native ON CONFLICT clause.

postgres-upsert.sql
INSERT INTO customers (customer_id, full_name, email)
VALUES ($1, $2, $3)
ON CONFLICT (customer_id)
DO UPDATE SET
  full_name = EXCLUDED.full_name,
  email = EXCLUDED.email;

ON CONFLICT

Tells Postgres: "if a row with this customer_id already exists, don't error — update it instead."

EXCLUDED

Refers to the row that would have been inserted, letting you pull its values into the update.

Requirement

This only works if customer_id has a unique constraint or index at the database level. Upsert cannot function on a column with no uniqueness guarantee — Postgres has no way to detect "conflict" otherwise.

Upsert

Insert if new, Update if exists. Use when the incoming data may contain both new and existing records.

Example: CRM sync — 5,000 contacts, some new, most already exist.
Skip on Conflict

Lighter-weight alternative. If a row violates a unique constraint, silently skip it instead of throwing an error.

Example: Preventing duplicate charges — never insert the same order_id twice.
Section 05 — Critical

What Happens When a Write Fails Mid-Batch

Suppose 1,000 customer records are being inserted, and record #612 violates a unique constraint on email. Here is the exact failure state for each batching mode.

Independently + Continue on Fail

  • • Records 1–611 → committed individually, exist and are safe
  • • Record 612 → fails, marked with error in output
  • • Records 613–1000 → still attempted, each succeeds or fails on its own
Best for: Batch processing where partial success is acceptable

Single Query

  • • Combined query may fail as a whole
  • • Potentially rejects the entire batch rather than just record 612
  • • Behavior depends on how the query was constructed
Best for: Speed when all items are known-good

Transaction

  • • Postgres automatically issues ROLLBACK
  • • None of the 1,000 rows are committed
  • • Database is left exactly as it was before execution started
Best for: Financial data, invoices — any case where partial writes corrupt state

Skip on Conflict

Database silently ignores duplicate row and continues. No error propagates to n8n.

Continue on Fail

Record 612 marked as failed, but records 613–1000 still get attempted.

Upsert Instead

If the "duplicate" should update existing record, Upsert removes the failure mode entirely.

Section 06

Real-World Production Patterns

Five scenarios where the difference between a hobbyist and a consultant is sharpest.

CRM Sync — 5,000 Contacts Daily

Some brand-new, most already exist. Upsert keyed on email handles both cases in a single operation, correctly, every time.

Financial Ledger Integrity

12 related rows (invoice + line items) must be atomic. Transaction batching: if any row fails, none are written.

Preventing Duplicate Charges

Webhook fires twice due to network retry. Unique constraint + Skip on Conflict makes duplication structurally impossible.

Security Incident Prevention

Support-ticket intake takes free-text customer input. Parameterization eliminates SQL injection entirely — not just "unlikely."

Nightly Reconciliation — 50,000 Rows

Single malformed row must not kill the other 49,999. Independently + Continue on Fail isolates failures per-item.

Section 07

The 90% vs The 10%

Anyone can wire up an Insert node. Very few n8n users can explain why a raw string-interpolated query is a liability.

The 90% (Button-Clickers) The 10% (You After Today)
Builds queries with SELECT * FROM users WHERE email = '${$json.email}' Uses $1/$2 parameterized queries — immune to injection regardless of input content
Uses "Insert" for everything, hits duplicate-key errors constantly Chooses Insert / Upsert / Update deliberately based on whether the record may already exist
Never touches Query Batching — leaves it at the surprising default Selects Single Query, Independently, or Transaction based on whether partial failure is acceptable
One bad row kills a 10,000-row nightly sync Configures Skip on Conflict or per-item isolation so one bad row doesn't take down the batch
Doesn't know what ON CONFLICT does Can read and write native Postgres upsert SQL from memory
