http-request-engine.js
// 1. CREDENTIAL RESOLUTION
const credential = await credentials.get('httpHeaderAuth');
// Decrypted from DB using instance encryption key
// NEVER appears in workflow JSON or execution logs

// 2. REQUEST BUILDING (axios under the hood)
const config = {
  url: 'https://api.stripe.com/v1/customers',
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${credential.token}` // Injected here
  },
  data: { name: 'Ahmed', email: 'ahmed@company.com' },
  timeout: 30000 // Default: none (production risk!)
};

// 3. TLS/SSL (Node.js tls module)
// Certificate validation automatic
// Self-signed certs fail unless "Allow Unauthorized Certs" is ON

// 4. HTTP EXECUTION
const response = await axios.request(config);

// 5. RESPONSE PROCESSING
// Status 4xx/5xx → throws error by default
// Body parsed via JSON.parse if content-type === application/json
// Output wrapped into n8n items array

Security Guarantee: API tokens never appear in workflow JSON exports, execution logs, browser URLs, or webhook responses. They live only in the encrypted database record.

Section 02

HTTP Methods — Protocol Semantics

Choosing the wrong method causes confusing API errors. The URL may be correct, but the method is wrong.

GET Retrieve

No body. Idempotent. Calling twice = same result.

• Fetch records
• Read state
• Search
POST Create

Has body. NOT idempotent. Calling twice creates twice.

• Create records
• Submit forms
• Trigger actions
PUT Replace

Full replacement. Idempotent. Omitting a field deletes it.

• Full updates
• Complete state
• Overwrite resource
PATCH Partial

Partial update. Idempotent. Preferred over PUT.

• Update specific fields
• Modify status
• Preferred method
DELETE Remove

Usually no body. Idempotent. Removes resource.

• Delete records
• Remove resources
• Archive data
HEAD Metadata

Same as GET but returns only headers. No body.

• Check existence
• Get metadata
• CORS preflight

Most Common Mistake

Using GET to send data (impossible — GET has no body) or using POST when the API expects PUT/PATCH (API returns 405 Method Not Allowed).

Section 04

Request Body Types

Choosing the wrong body type causes the API to reject your request with cryptic errors.

JSON

Content-Type: application/json

{"key": "value", "nested": {"field": 42}}

Use for: REST APIs (the vast majority)

Form URL Encoded

Content-Type: application/x-www-form-urlencoded

key=value&another=field&number=42

Use for: Legacy APIs, OAuth2 token endpoints

Multipart

Content-Type: multipart/form-data

------WebKitFormBoundary
Content-Disposition: form-data; name="file"

Use for: File uploads (binary data)

Section 05 — Critical

The Full Response Toggle

The equivalent of the webhook $json.body bug from Day 2, but for the HTTP Request node. The most commonly misunderstood setting.

When Full Response is ON, data is nested under $json.body

With Full Response OFF: {{ $json.id }} works. With Full Response ON: {{ $json.body.id }} is required. Confusing them is a frequent production bug.

Full Response: OFF
Default output
[
  {
    "json": {
      "id": 1,
      "name": "Ahmed",
      "email": "ahmed@company.com"
    }
  }
]
{{ $json.id }} ← Works
Full Response: ON
Wrapped output
[
  {
    "json": {
      "statusCode": 200,
      "headers": {
        "x-ratelimit-remaining": "98"
      },
      "body": {
        "id": 1,
        "name": "Ahmed"
      }
    }
  }
]
{{ $json.body.id }} ← Required

When to use Full Response ON

  • • API returns important data in headers (rate limits, pagination cursors)
  • • You need to check HTTP status code downstream
  • • Handle different responses based on status (200 vs 201 vs 204)
  • • Debugging — seeing the full response including headers
Section 06

Pagination — Three Built-in Strategies

n8n fetches multiple pages and merges all results into a single flat items array. Choose the wrong strategy and you lose records.

1

Offset / Page Number

Increment a numeric parameter

API Pattern
GET /users?page=1&per_page=100
GET /users?page=2&per_page=100
GET /users?page=3&per_page=100
// Stops when response empty or page > total
n8n Config
{
  "pagination": true,
  "mode": "updateParameter",
  "parameter": "page",
  "startValue": 1,
  "stopCondition": "{{ $response.body.data.length === 0 }}"
}
2

Cursor-Based

Extract next cursor from response body

API Pattern
GET /users
→ { "data": [...], "pagination": { "next_cursor": "abc123" } }

GET /users?cursor=abc123
→ { "data": [...], "pagination": { "next_cursor": null } }
// Stops when next_cursor is null
n8n Config
{
  "pagination": true,
  "mode": "updateParameter",
  "parameter": "cursor",
  "value": "{{ $response.body.pagination.next_cursor }}",
  "stopCondition": "{{ !$response.body.pagination.has_next }}"
}
3

Link Header

Extract next URL from response headers

API Pattern (GitHub)
HTTP/1.1 200 OK
Link: <https://api.github.com/repos?page=2>; rel="next",
      <https://api.github.com/repos?page=5>; rel="last"

// Stops when no "next" link in header
n8n Config
{
  "pagination": true,
  "mode": "responseContainsNextUrl",
  "nextUrl": "{{ $response.headers.link.match(/<([^>]+)>; rel=\"next\"/)?.[1] }}"
}

Result: After pagination completes, the output is a single flat items array containing ALL records from ALL pages — as if they came from one response.

Section 07

Production Error Handling Matrix

By default, any 4xx or 5xx stops the workflow. Here is how to handle each status code correctly.

Status Meaning Strategy Why
429 Too Many Requests Retry with delay Transient — waiting resolves it
503 Service Unavailable Retry with delay Transient — server will recover
400 Bad Request Continue on Error Retrying won't fix bad data
401 Unauthorized Alert immediately Infrastructure issue — don't retry
404 Not Found Continue on Error Record may have been deleted
500 Internal Server Error Retry once, then alert May be transient or permanent

Retry on Fail

Configurable retries with wait interval. Correct for 429, 503, 502, 504. Wrong for 400, 401, 403, 404.

Continue on Error

Marks failed items with error but continues processing. Correct for batch processing where one bad record shouldn't stop 999 good ones.

Response Code Routing

Route different status codes to different paths. 200-299 → success. 400-499 → client error. 500-599 → server error.
