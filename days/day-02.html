Section 03 — Critical

The Most Common Mistake in n8n

This single mistake kills more workflows than anything else. Read it three times.

It's $json.body — not $json directly

In every other node, $json holds your data fields directly. But the Webhook node wraps the entire HTTP request — your actual data is one level deeper, inside $json.body.

Webhook Node Output Structure 
[
  {
    "json": {
      "headers": { "content-type", "authorization", "x-api-key" },
      "params": {},
      "query": { "source": "mobile-app" },
      "body": {
        "order_id": "ORD-992",
        "customer_name": "Ahmed",
        "total_amount": 499.99
      }, ← YOUR DATA IS HERE
      "webhookUrl": "https://.../webhook/order-api",
      "executionMode": "production"
    }
  }
]
What You Want Wrong — Returns Undefined Correct Expression
Body: order_id {{ $json.order_id }} {{ $json.body.order_id }}
Body: customer_name {{ $json.customer_name }} {{ $json.body.customer_name }}
Header: x-api-key {{ $json['x-api-key'] }} {{ $json.headers['x-api-key'] }}
Header: Authorization {{ $json.authorization }} {{ $json.headers.authorization }}
Query: source {{ $json.source }} {{ $json.query.source }}
Section 04

Test URL vs Production URL

A critical operational distinction — sending data to the wrong URL causes random silent failures in production.

Property Test URL Production URL
Format .../webhook-test/{path} .../webhook/{path}
Active When Editor is open AND "Listen for Test Event" was clicked Workflow toggle is ON (active)
Execution Storage Execution log in editor only Per your retention settings
Use Case Development only Production — always
Danger of Misuse Partner sends to Test URL → fires only if editor is open → random silent failures Using during dev → creates real executions → may trigger real side effects

The Production Rule

Once you hand a webhook URL to an external system, it must always be the Production URL. The workflow must be toggled Active. This is non-negotiable.

Section 05

Three Webhook Response Modes

Each mode fundamentally changes how the caller interacts with n8n. Picking the wrong mode breaks the integration.

1

Immediately

fire-and-forget
POST → n8n → 200 OK instantly Workflow runs in background

When to use: The caller doesn't need a result. Log pipelines, analytics events, audit trails.

2

When Last Node Finishes

synchronous
POST → n8n → Runs all nodes Returns last node output

When to use: The caller needs the processed result back. Synchronous API patterns.

3

Respond to Webhook Node

professional pattern
POST → n8n → Some nodes execute Custom response → rest in background

When to use: You need a custom JSON response AND want it sent before heavy processing completes. The professional pattern for building APIs in n8n.

Section 06

Cron Expression Syntax — The 5 Fields

A Cron expression has exactly 5 fields. Each controls a different time dimension. Memorize these expressions.

*
Minute
0 – 59
*
Hour
0 – 23
*
Day of Month
1 – 31
*
Month
1 – 12
*
Day of Week
0 – 7 (0/7=Sun)
* = every value    */n = every n units    n-m = range    n,m = specific values
0 8 * * * Every day at 8:00 AM
0 8 * * 1-5 Weekdays (Mon-Fri) at 8:00 AM
*/15 * * * * Every 15 minutes
0 0 * * 0 Every Sunday at midnight
0 6 1 * * 1st of every month at 6:00 AM
0 */6 * * * Every 6 hours
0 9,18 * * * Twice daily: 9 AM and 6 PM
59 23 * * 0 Every Sunday at 23:59

What happens on n8n restart?

The scheduler re-reads all active workflows and re-registers all cron jobs. If a job was scheduled for 08:00 and n8n restarts at 07:59 — the 08:00 execution fires normally. But past-due executions are NOT backfilled — if n8n was down from 07:00 to 09:00, the 08:00 job is silently skipped.

Section 07

Push vs Polling — Two Different Architectures

App Event triggers use one of these two architectures. Understanding the difference determines your integration choice.

Push Architecture

The external service detects an event and sends an instant POST to your n8n webhook URL. n8n does nothing proactively.

  • External service detects event
  • Instant POST to n8n webhook URL
  • Instant response — zero latency
Services using this pattern:
Stripe GitHub WhatsApp Slack

Polling Architecture

n8n sends periodic requests to the external service, compares the response against a stored cursor, and passes only new items forward.

  • n8n sends periodic GET with lastCursor
  • Filter: only items with ID > lastCursor
  • Update cursor → store new lastCursor
  • Pass only new items to workflow
Services using this pattern:
Gmail (some modes) Legacy services
Deduplication: n8n stores the highest event ID or timestamp seen and only processes records newer than that value — preventing the same event from being processed twice.
