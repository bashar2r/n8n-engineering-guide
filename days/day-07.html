Section 01

The Three "On Error" Modes

Every node in n8n has an On Error setting with exactly three options. Choosing the wrong one silently changes your workflow's failure behavior.

STOP WORKFLOW Default
  • • Halts the entire workflow the instant this node errors
  • • No further nodes execute
  • • Execution status becomes ERROR
  • • If an Error Workflow is configured, it fires
Use when: Any failure is unacceptable — financial transactions, critical API calls
CONTINUE Least Safe
  • • Proceeds to the next node despite the error
  • • Uses the LAST VALID DATA — stale/unchanged
  • • No visible marker that anything went wrong downstream
  • • Failures become invisible
⚠ Dangerous: Silent data corruption. Use only when you explicitly track failures downstream.
CONTINUE (ERROR OUTPUT) Production Choice
  • • Node gains a SECOND output connector (error output)
  • • Failed items routed there instead of crashing
  • • Successful items → normal output; failed items → error output
  • • Wire each to different logic
Use when: One bad row shouldn't kill 999 good ones — batch processing, data pipelines
Section 02 — Critical

The Retry On Fail Gotcha

Today's equivalent of Day 2's $json.body and Day 5's parameterized queries — the mistake that looks fine in testing and fails invisibly in production.

Retry On Fail is silently bypassed unless On Error = "Stop Workflow"

If you set On Error to Continue or Continue (error output) — the retry counter can be bypassed entirely. The node fails once, immediately routes to the error path or continues, and never attempts try #2 or #3, even though the UI shows "3 tries" configured.

SILENTLY BROKEN
node-config.json
{
  "retryOnFail": true,
  "maxTries": 3,
  "waitBetweenTries": 2000,
  "onError": "continueErrorOutput"
  // ↑ Retry is BYPASSED — attempts 2 & 3 NEVER happen
}

What happens:

Attempt 1 fails → routes to error output immediately → Attempts 2 & 3 never happen

CORRECT
node-config.json
{
  "retryOnFail": true,
  "maxTries": 3,
  "waitBetweenTries": 2000,
  "onError": "stopWorkflow"
  // ↑ Retry works as configured
}

What happens:

Attempt 1 fails → waits 2s → Attempt 2 → waits 2s → Attempt 3 → then Error Workflow fires

Verification Rule

Never assume a Retry + On Error combination works as configured. Deliberately test it by pointing the node at something that fails predictably (a dead URL, a wrong port) and counting the actual attempts in the execution log.

Section 03

Stop And Error — Deliberate Failure

Force a workflow to fail on your own terms, with your own message, at the exact point you've verified something is wrong — rather than letting a downstream node fail confusingly three steps later.

Error Message

Throw a simple custom text message. Useful for human-readable alerts.

"Order rejected — missing order_id
 or invalid amount: {{ $json.order_id }}"

Error Object

Throw a full JSON object. Useful when your Error Workflow needs to branch logic based on structured data.

{
  "type": "validation_failed",
  "field": "order_id",
  "reason": "missing",
  "severity": "critical"
}

Hard Constraint

Stop And Error can only be used as the LAST node in a workflow (or branch). It doesn't make sense to connect anything after it — by definition, it terminates that execution path with a failure.

Example: Validation Gate Before Database Write

Set Node
Shape order data
IF Node
order_id exists AND amount > 0?
TRUE → Postgres Insert
Data is valid, proceed to write
FALSE → Stop And Error
Fail immediately with clear message
Section 04

Error Trigger + Error Workflow

n8n's workflow-level safety net. A separate workflow that fires automatically when any monitored workflow fails.

Setup Steps

1

Create a new, separate workflow. Add the Error Trigger node as its first node.

2

Save it — name it clearly: Error Handler — Order Pipeline

3

In the monitored workflow: Options (⋮) → Settings → Error Workflow → select your handler.

4

Save the monitored workflow.

No Activation Needed

You do not need to toggle the Error Workflow "Active." Unlike Schedule or Webhook workflows, a workflow whose first node is Error Trigger doesn't need normal activation — n8n invokes it directly and internally when the monitored workflow fails.

Self-Reference Default

If a workflow itself contains an Error Trigger node, it defaults to using itself as its own Error Workflow unless you point it elsewhere.

Section 05

The Exact Payload Structure

The Error Trigger receives a specific JSON payload. Its shape changes depending on where the failure happened.

error-trigger-payload.json
[
  {
    "execution": {
      "id": "231",
      "url": "https://your-instance.app.n8n.cloud/execution/231",
      "retryOf": "34",
      "error": {
        "message": "duplicate key value violates unique constraint \"customers_email_key\"",
        "stack": "..."
      },
      "lastNodeExecuted": "Upsert Customer",
      "mode": "trigger"
    },
    "workflow": {
      "id": "17",
      "name": "Customer Sync Pipeline"
    }
  }
]

execution.id & execution.url

Only exist if the execution was saved to the database. Absent entirely if the failure happened in the trigger node itself — the workflow never truly executed.

execution.retryOf

Appears only when this run is a retry of a previously failed execution. This is your signal that "this is a repeat attempt" — critical for idempotency logic.

Trigger vs Mid-Workflow

If failure originates in the trigger node, the payload shape changes — less detail under execution{}, more under a trigger{} object instead.

Building the Notification Message

🔴 Workflow Failed: {{ $json.workflow.name }}
Node: {{ $json.execution.lastNodeExecuted }}
Error: {{ $json.execution.error.message }}
Execution: {{ $json.execution.url }}
Retry of previous failure: {{ $json.execution.retryOf ? 'Yes' : 'No' }}
Section 06 — Critical

The Testing-Discipline Gotcha

Confirmed directly in the docs: manually clicking "Execute Workflow" will never trigger your Error Workflow. Easy to lose an afternoon to if you don't know it up front.

The Error Trigger only fires for automatic executions

Webhook, schedule, polling — these trigger the Error Workflow. Manually clicking "Execute Workflow" in the editor, even on a workflow that fails, will never trigger the configured Error Workflow. If you're testing your error-handling setup by manually running the monitored workflow and wondering why no Slack message arrives — nothing is broken.

WRONG WAY
  • • Open workflow in editor
  • • Click "Execute Workflow"
  • • Node fails
  • • No Slack message arrives
  • • Spend 2 hours debugging
RIGHT WAY
  • • Fire the actual webhook (curl, Postman)
  • • Or wait for the actual schedule
  • • Or temporarily lower schedule interval
  • • Node fails
  • • Slack message arrives instantly
Section 07

Idempotency — Safe to Run Twice

The core question every production write operation must answer: if this exact execution ran a second time, would anything bad happen?

NOT IDEMPOTENT
-- Postgres "Insert" with NO unique constraint
INSERT INTO orders (order_id, amount)
VALUES ('ORD-001', 299.99);
  • • Webhook retries due to network blip
  • • Same order inserted TWICE
  • • Customer charged twice
  • • Inventory decremented twice
Dangerous: Duplicate delivery = double effect
IDEMPOTENT
-- UNIQUE constraint on order_id
-- + Skip on Conflict OR Upsert
INSERT INTO orders (order_id, amount)
VALUES ('ORD-001', 299.99)
ON CONFLICT (order_id) DO NOTHING;
  • • Webhook retries due to network blip
  • • Duplicate delivery either does nothing
  • • Or safely overwrites with identical data
  • • No double effect
Safe: Duplicate delivery = no harm

The Rule

Every node that writes to an external system — database, CRM, payment gateway, email send — should be evaluated with the question: "is it safe for n8n to run this exact step twice?" If the answer is no, that's a design defect to fix before the workflow goes anywhere near production traffic.

execution.retryOf

The execution.retryOf field exists precisely because retries are not a rare edge case — they're something n8n assumes will happen and gives you the data to detect. Use it in your Error Workflow to distinguish first failures from retries.

Section 08 — Quiz

Test Your Understanding

Three quick questions testing deep comprehension of error handling architecture.

1. You configure "Retry On Fail: 3 tries" but set On Error to "Continue (using error output)." What happens?

All 3 retries execute, then failed items route to error output
Attempt 1 fails, immediately routes to error output — retries 2 & 3 never happen
The workflow stops with an error saying the settings are incompatible
n8n automatically changes On Error to "Stop Workflow" to allow retries

2. You want to test that your Error Workflow sends a Slack alert. What's the correct way?

Click "Execute Workflow" in the editor and watch for the Slack message
Toggle the Error Workflow to "Active" and then click Execute
Fire the actual webhook or wait for the actual schedule — manual execution never triggers Error Workflows
Add an Error Trigger node to the main workflow itself

3. A payment webhook fires twice due to a network retry. Which design prevents double-charging?

Use "Continue" on the Insert node so the second webhook is ignored
Add a UNIQUE constraint on order_id + use Skip on Conflict or Upsert
Set "Retry On Fail" to 0 so the webhook handler never retries
Use a Stop And Error node before the payment node to block duplicates
