Critical Rule

If a field does not exist, $json.missingField returns undefined, not an error. But $json.missingObject.field — accessing a property on undefined — throws a runtime error and fails the node. This is the root cause of most Code node production crashes.

Section 02

Set / Edit Fields Node — Complete Reference

The Set node has two operating modes. The single most important toggle is "Keep Only Specified Fields" — it determines whether you pass a clean payload or a bloated one downstream.

Mode 1: Manual

You define each output field explicitly. The most common mode for day-to-day transformations.

Most Common

Mode 2: Raw JSON

You write a complete JSON object. Used when the output shape is itself dynamic.

Dynamic Shape
"Keep Only Specified" OFF vs ON 
// OFF (default): ALL original fields PLUS your new ones
Input:  { id: 1, name: "Ahmed", salary: 1200, address: {...}, company: {...} }
Fields: employee_id = {{ $json.id }}, display = {{ $json.name }}
Output: { id: 1, name: "Ahmed", salary: 1200, address: {...}, company: {...},
          employee_id: 1, display: "Ahmed" }

// ON: ONLY your defined fields — everything else stripped
Input:  { id: 1, name: "Ahmed", salary: 1200, address: {...}, company: {...} }
Fields: employee_id = {{ $json.id }}, display = {{ $json.name }}
Output: { employee_id: 1, display: "Ahmed" }
// Everything else is GONE

Production Rule

In production pipelines with real API responses (often 20–50 fields), always use "Keep Only Specified" to pass only what downstream nodes need. Sending unnecessary data wastes memory, inflates execution logs, and causes bugs when unexpected field names collide with downstream nodes' expected inputs.

Section 03 — The Ceiling Remover

Code Node — The Two Execution Modes

This is the most architecturally significant distinction in the Code node. Getting it wrong produces subtle, hard-to-diagnose bugs. Every no-code tool hits a wall — the Code node eliminates that wall entirely.

Mode 1: Run Once for Each Item

Executes N times for N items. Each item is processed independently. Cannot aggregate across items. 

// Executes N times for N items
// $input.item = the current item
// Must return a SINGLE item object

const item = $input.item.json;

return {
  json: {
    ...item,
    computed_field: item.price * item.quantity
  }
};

// Use when: each item processed independently
// Cannot do: cross-item sums, counts, averages
When to use:
Per-item transforms Field calculations String manipulation

Mode 2: Run Once for All Items

Executes ONCE regardless of item count. Full array is available. Cross-item computation is now possible. 

// Executes ONCE for all items
// $input.all() = the entire array
// Must return an ARRAY of items

const items = $input.all();

// Cross-item computation
const grandTotal = items.reduce(
  (sum, i) => sum + (i.json.price * i.json.quantity), 0
);

const processed = items.map(item => ({
  json: {
    ...item.json,
    subtotal: item.json.price * item.json.quantity,
    grand_total: grandTotal,
    percentage: Math.round(
      (item.json.price * item.json.quantity) / grandTotal * 10000
    ) / 100
  }
}));

return processed;

// Use when: aggregating across ALL items
When to use:
Totals & averages Deduplication Sorting Cross-item logic

Mandatory Selection Rule

Do you need data from item A while processing item B? → Run Once for All Items
Does each item's transformation depend only on itself? → Run Once for Each Item

$input Methods Available in Both Modes 
$input.item              // Current item ("per item" mode)
$input.all()             // Full array of all input items
$input.first()           // First item (index 0)
$input.last()            // Last item
$input.itemMatching(0)   // Item at specific index

// Reference other nodes:
$('HTTP Request').all()    // All items from named node
$('HTTP Request').first()  // First item from named node
$('HTTP Request').item     // Paired item (same index)
Section 04

Filter Node — Deletion, Not Routing

The Filter node evaluates a condition for every item. Items where the condition is true proceed. Items where the condition is false are permanently removed from the pipeline. Gone. Not sent anywhere.

Filter Behavior 
Input:  [A, B, C, D, E]
Condition: value > 100

Output: [A, C, D]     // B and E are GONE
                        // Not sent anywhere, not error-handled
Multi-Condition Combinations 
// AND combinator: ALL conditions must be true
condition 1: active === true
condition 2: salary > 1000
condition 3: department !== 'Temp'
// Only items satisfying ALL three pass

// OR combinator: ANY condition being true passes the item
condition 1: department === 'Engineering'
condition 2: salary > 2000
// Items in Engineering OR earning > 2000 pass

When to Use Filter

When you genuinely want to discard records: invalid data, already-processed records, records outside date range, test records in production.

Section 05

IF Node — Routing, Not Deletion

The IF node evaluates a condition for every item, then routes each item to one of two outputs. Every item continues — to a different branch. Total items in = total items out (across both branches).

IF Behavior 
Input:  [A, B, C, D, E]
Condition: value > 100

True output:  [A, C, D]     → branch processing high values
False output: [B, E]        → branch processing low values

Total: 5 in = 5 out (across both branches)
Situation Use Filter Use IF
Invalid records — discard entirely ✅
Records needing different processing paths ✅
Filtering test data from production data ✅
VIP customers vs standard customers ✅
Records missing required fields ✅
Large vs small orders ✅

The Fatal Confusion

Filter deletes items. IF routes items. These are architecturally opposite behaviors. Using them interchangeably is the #1 cause of "missing data" bugs in production workflows.

Section 07 — Production Safety

Defensive Coding Patterns in the Code Node

The Code node crashes in production with 500 items when practitioners don't validate. These patterns prevent that.

Pattern 1: Null-Check Before Access 
// ❌ DANGEROUS: Crashes if customer is null
const city = item.json.customer.address.city;

// ✅ SAFE: Graceful fallback
const city = item.json.customer?.address?.city || 'Unknown';

// ✅ SAFER: Explicit validation
const customer = item.json.customer;
const city = customer && customer.address 
  ? customer.address.city 
  : 'Unknown';
Pattern 2: Type Validation 
// ❌ DANGEROUS: String concatenation instead of math
const total = item.json.price + item.json.tax;  // "100" + "15" = "10015"

// ✅ SAFE: Explicit casting
const total = Number(item.json.price) + Number(item.json.tax);  // 115

// ✅ SAFER: Validate before casting
const price = typeof item.json.price === 'number' 
  ? item.json.price 
  : parseFloat(item.json.price) || 0;
Pattern 3: Array Safety 
// ❌ DANGEROUS: Crashes if tags is not an array
const firstTag = item.json.tags[0];

// ✅ SAFE: Always check array
const tags = Array.isArray(item.json.tags) ? item.json.tags : [];
const firstTag = tags[0] || 'none';

// ✅ SAFER: Default empty array
const { tags = [] } = item.json;
const isPremium = tags.includes('premium');

Code Node — Run Once for All Items 
const items = $input.all();

// Compute grand total first
const grandTotal = items.reduce((sum, item) => {
  const amount = typeof item.json.amount === 'number' 
    ? item.json.amount 
    : parseFloat(item.json.amount) || 0;
  return sum + amount;
}, 0);

const processed = items.map(item => {
  const raw = item.json;

  // Defensive date normalization
  const dateStr = raw.date || raw.invoice_date || raw.created_at;
  const normalizedDate = dateStr 
    ? new Date(dateStr).toISOString() 
    : 'invalid-date';

  const amount = Number(raw.amount) || 0;
  const tax = amount * 0.15;

  return {
    json: {
      invoice_id: raw.id || 'UNKNOWN',
      customer: raw.customer?.name || 'Guest',
      amount: amount,
      tax: Math.round(tax * 100) / 100,
      total: Math.round((amount + tax) * 100) / 100,
      date: normalizedDate,
      percentage_of_total: grandTotal > 0 
        ? Math.round((amount / grandTotal) * 10000) / 100 
        : 0
    }
  };
});

return processed;
